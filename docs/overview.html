Architecture overview of vue-pdf-live-render, a measurement-driven live-preview pagination engine for Vue and Tailwind documents

Architecture overview

vue-pdf-live-render

A live-preview pagination engine for Vue 3 + Tailwind business documents — invoices, offers, delivery notes. Word-style multi-page layout with running headers/footers, page numbers and an instant preview of real page breaks. Content stays real Vue components; the final PDF is still printed by Puppeteer from the same DOM.

The core insight

Four browser realities decide the whole design. The synthesis of them is: render once, measure, project — never relocate.

01 · no pouring

You can't flow content into page boxes. CSS Regions are dead in Chromium and element() never shipped. You must paginate by measurement + DOM splitting.

02 · screen is blind

break-* is inert on screen. It only acts inside a fragmentation container. So we measure real geometry ourselves and re-emit our decisions as forced breaks for print.

03 · avoid is a lie

break-inside:avoid is a hint. LayoutNG ignores it under pressure. For the PDF we resolve every keep/orphan/widow rule and bake authoritative break-before:page.

04 · don't fight Vue

Moving live DOM breaks Vue. It patches in place against a fixed anchor. So instances are never cloned or relocated — a component spanning pages is windowed, not split.

Why not the existing tools

paged.js / Vivliostyle

Built for books/prose. Clones & moves content (loses component state), re-lays out the whole document on any change (no live reactivity), and can deadlock on oversized content. We keep their two good ideas — browser-measured layout + resumable break tokens — and fix the rest.

@react-pdf / pdfmake

A separate layout engine (Yoga) with its own style dialect. Clean pagination, but you lose CSS/Tailwind fidelity and re-author every document. We measure the real rendered DOM instead, so there's no dialect to drift from.

Designed via a multi-agent pass — 6 research probes → 3 competing architectures → a 3-lens judge panel → synthesis. The candidates converged ~90%; the chosen design takes the cleanest core, grafts in continuation-fragment rendering for repeated headers, and adopts the cross-cutting agreements.
